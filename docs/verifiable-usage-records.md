"at least one form of verifiable usage record (test logs, user records,
or sample input/output)". This document lists every artifact under
`verifiable_usage_records/` and explains what each one proves.

## How to regenerate

Every artifact in this directory is regeneratable. From the repo root:

```bash
# 1. Install (3 commands — see INSTALL.md)
git clone https://github.com/visitsolo1/Joulyzer.git
cd Joulyzer
pip install -e ".[test]"

# 2. Run the tests (text log)
python -m pytest tests/ -v | tee verifiable_usage_records/pytest-2026-06-24.txt

# 3. Run the live usage harness (writes everything else)
python scripts/live_usage_harness.py

# 4. Or one-shot — runs tests, CLI runs, integration example, MCP demo,
#    live harness, tool schema, MCP manifest, and prints a summary.
bash scripts/verify_submission.sh
```

## Why these count as "verifiable"

* **They are reproducible.** Every script is committed under `scripts/`.
  A judge can `pip install -e ".[test]"`, run `bash scripts/verify_submission.sh`,
  and watch every artifact get regenerated in ~3 seconds.
* **They exercise the same code paths a real agent would.** The CLI
  subprocess, in-process function call, and JSON-RPC MCP subprocess are
  the three realistic ways an AI agent consumes joulyzer; the demo run
  is joulyzer's own end-to-end example; the live Bitget calls prove the
  infra talks to a real exchange.
* **They mix synthetic and real data.** Generated journals make the
  structure auditable; the live Bitget `/public/time` call proves the
  network path actually works against `api.bitget.com`.
* **Honest failures are recorded, not hidden.** The harness emits an
  `error` record (status, error message, latency) instead of aborting,
  so judges see exactly what happened if a network call or a missing
  file occurs.
